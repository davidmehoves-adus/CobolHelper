# Copybooks

## Description

Copybooks are reusable source text files that are included into COBOL programs at compile time via the COPY statement. They serve as the primary mechanism for sharing data definitions, record layouts, constant declarations, and common code paragraphs across multiple programs in an enterprise COBOL environment. Understanding copybooks is essential for working in any production COBOL system because they enforce consistency, reduce duplication, and form the backbone of interface contracts between programs, files, databases, and messaging systems.

## Core Concepts

### What Is a Copybook

A copybook is a file (or partitioned data set member, on mainframes) containing COBOL source text that is designed to be included in one or more COBOL programs at compile time. Copybooks are not compiled independently; instead, the compiler reads them inline wherever a COPY statement references them. The included text becomes part of the compiling program as if it had been typed directly into the source.

Copybooks typically contain:

- **Data descriptions** (record layouts, working-storage groups, file descriptors)
- **Constant definitions** (88-level condition names, literal values)
- **Paragraph or section code** (common processing logic)
- **EXEC SQL INCLUDE members** (DB2 host variable declarations generated by DCLGEN)
- **Screen definitions** (SCREEN SECTION entries in interactive environments)

The COBOL standard defines the COPY statement in the COBOL source text manipulation section of the language. Every major COBOL compiler -- IBM Enterprise COBOL, Micro Focus Visual COBOL, GnuCOBOL, Fujitsu NetCOBOL, and others -- supports it.

### Copybook vs Source Member

The terms "copybook" and "source member" are sometimes used interchangeably, but they have distinct meanings:

| Term | Meaning |
|------|---------|
| **Source member** | Any file or PDS member containing COBOL source code, including main programs. |
| **Copybook** | A source member specifically intended for inclusion via COPY, not compiled on its own. |

A copybook typically does not contain full program structure (IDENTIFICATION DIVISION, etc.). It contains fragments -- a group of data items, a set of paragraphs, or a block of declarations -- that slot into a specific location within the including program.

On z/OS mainframes, copybooks are stored as members of partitioned data sets (PDS or PDSE). On distributed platforms, they are ordinary files in the file system, often with extensions like `.cpy`, `.cbl`, `.cob`, or `.copy`, though this varies by shop and compiler.

### How Inclusion Works

When the compiler encounters a COPY statement, it:

1. Locates the named copybook in the specified (or default) library.
2. Reads the entire text of the copybook.
3. Applies any REPLACING operands to transform the text.
4. Inserts the resulting text into the compilation stream at the point of the COPY statement.
5. Continues compiling as if the inserted text were part of the original source.

This is a purely textual inclusion mechanism. The compiler does not evaluate the copybook in isolation; it simply splices the text into the compilation unit. As a result, a copybook's content must be syntactically valid in the context where it is included.

### Library Organization

Copybooks are organized into libraries. The way libraries are specified depends on the platform:

**z/OS (Mainframe)**

On z/OS, copybook libraries are partitioned data sets specified via JCL:
- The `SYSLIB` DD statement in the compile step defines the default copybook search path.
- Multiple concatenated data sets can be listed under `SYSLIB`, searched in order.
- The `IN` or `OF` clause on the COPY statement can direct the compiler to a specific DD name.

**Distributed Platforms (Micro Focus, GnuCOBOL, etc.)**

On distributed platforms, copybook libraries are directories:
- Compiler options or environment variables specify the search path (e.g., `-I` flag, `COBCPY` environment variable, or IDE project settings).
- Some compilers allow specifying the full or relative path to the copybook.

**Typical library organization by function:**

- `COPYLIB` -- General-purpose copybooks (common record layouts, constants)
- `DCLGENLIB` -- DB2 DCLGEN output (host variable declarations and SQL INCLUDE members)
- `CICSLIB` or `CICS.COPYLIB` -- CICS-related copybooks (DFHCOMMAREA layouts, BMS map copybooks)
- `MQLIB` -- MQ Series message structure copybooks
- `APPLIB` -- Application-specific shared structures

### Naming Conventions

Copybook naming conventions vary across installations, but common patterns include:

**Prefix-based naming:**
- A 2-3 character application or subsystem prefix followed by a descriptive root.
- Example: `CUSTREC` (customer record), `ACCTLAY` (account layout), `ORDHDR` (order header).

**Suffix-based typing:**
- Some shops use suffixes to indicate the content type:
  - `xxxREC` or `xxxLAY` -- record layout
  - `xxxCPY` -- general copybook
  - `xxxDCL` -- DCLGEN output
  - `xxxWSS` -- working-storage items
  - `xxxPAR` -- paragraph include

**Length constraints:**
- On z/OS, PDS member names are limited to 8 characters. This is the most common constraint that drives abbreviated naming.
- On distributed platforms, longer names are available, and shops often adopt more descriptive names.

**General best practices:**
- Keep names meaningful within the 8-character limit where applicable.
- Use a consistent prefix for all copybooks belonging to the same application.
- Maintain a naming standard document so developers can locate copybooks predictably.
- Avoid names that collide with compiler-supplied or system copybooks.

## Syntax & Examples

### Basic COPY Statement

The simplest form of the COPY statement names a copybook to include:

```cobol
       COPY CUSTREC.
```

This tells the compiler to find the member `CUSTREC` in the default library (SYSLIB on z/OS, or the configured search path on other platforms) and insert its contents at this point.

The COPY statement can appear anywhere a COBOL statement or clause can appear. It is most commonly placed in:
- The `DATA DIVISION` (WORKING-STORAGE SECTION, FILE SECTION, LINKAGE SECTION)
- The `PROCEDURE DIVISION` (to include paragraphs or sections)
- The `ENVIRONMENT DIVISION` (less common, but valid for FILE-CONTROL entries)

**Full minimal example:**

```cobol
       IDENTIFICATION DIVISION.
       PROGRAM-ID. CUSTPROG.

       DATA DIVISION.
       WORKING-STORAGE SECTION.
       COPY CUSTREC.

       PROCEDURE DIVISION.
           DISPLAY WS-CUST-NAME
           STOP RUN.
```

Where the copybook `CUSTREC` might contain:

```cobol
       01  WS-CUSTOMER-RECORD.
           05  WS-CUST-ID          PIC 9(08).
           05  WS-CUST-NAME        PIC X(30).
           05  WS-CUST-ADDR        PIC X(50).
           05  WS-CUST-BALANCE     PIC S9(7)V99 COMP-3.
```

### The IN / OF Clause

The `IN` or `OF` clause directs the compiler to search a specific library for the copybook. `IN` and `OF` are interchangeable.

```cobol
       COPY CUSTREC IN CUSTLIB.
       COPY CUSTREC OF CUSTLIB.
```

On z/OS, `CUSTLIB` refers to a DD name in the compile JCL. On distributed platforms, it typically refers to a directory or library name configured in the compiler settings.

**Example with JCL (z/OS):**

```
//CUSTLIB  DD DSN=PROD.CUST.COPYLIB,DISP=SHR
```

The compiler searches `PROD.CUST.COPYLIB` for member `CUSTREC` when it encounters `COPY CUSTREC IN CUSTLIB`.

### The REPLACING Clause

The `REPLACING` clause performs textual substitution on the copybook text as it is included. This is one of the most powerful and frequently used features of the COPY statement.

**Syntax:**

```cobol
       COPY copybook-name
           REPLACING operand-1 BY operand-2
                     operand-3 BY operand-4
                     ...
```

Each operand can be one of the following types:

| Operand Type | Delimiters | Example |
|---|---|---|
| Pseudo-text | `==` delimiters | `==:PREFIX:== BY ==WS-CUST-==` |
| Identifier | bare word | `FILLER BY WS-DETAIL` |
| Literal | quoted string | `"OLD" BY "NEW"` |
| COBOL word | bare word | `OLD-NAME BY NEW-NAME` |

**Pseudo-text replacing (most common and most flexible):**

Pseudo-text operands are enclosed in `==` delimiters and can match partial words, multiple words, or even empty text:

```cobol
       COPY CUSTREC
           REPLACING ==:TAG:== BY ==WS-CUST-==.
```

If the copybook `CUSTREC` contains:

```cobol
       01  :TAG:RECORD.
           05  :TAG:ID             PIC 9(08).
           05  :TAG:NAME           PIC X(30).
           05  :TAG:ADDR           PIC X(50).
           05  :TAG:BALANCE        PIC S9(7)V99 COMP-3.
```

After replacing, the compiler sees:

```cobol
       01  WS-CUST-RECORD.
           05  WS-CUST-ID         PIC 9(08).
           05  WS-CUST-NAME       PIC X(30).
           05  WS-CUST-ADDR       PIC X(50).
           05  WS-CUST-BALANCE    PIC S9(7)V99 COMP-3.
```

This pattern allows a single copybook to be reused with different prefixes in different programs.

**Multiple REPLACING pairs:**

```cobol
       COPY TRANREC
           REPLACING ==:PFX:==   BY ==TX-==
                     ==:RECNM:== BY ==TRANSACTION-RECORD==
                     ==:LEN:==   BY ==256==.
```

**Identifier replacing:**

```cobol
       COPY CUSTREC
           REPLACING CUSTOMER-RECORD BY MY-CUSTOMER-RECORD.
```

This replaces whole-word occurrences of `CUSTOMER-RECORD` with `MY-CUSTOMER-RECORD`. Unlike pseudo-text, identifier replacing only matches complete words.

**Literal replacing:**

```cobol
       COPY STATUSCD
           REPLACING "ACTIVE" BY "A"
                     "INACTIVE" BY "I".
```

### REPLACE Statement vs REPLACING Clause

COBOL has two distinct text-replacement mechanisms that are sometimes confused:

| Feature | REPLACING Clause | REPLACE Statement |
|---|---|---|
| **Scope** | Applies only to the copybook being included | Applies to all subsequent source text |
| **Where used** | On the COPY statement | Standalone statement in source |
| **Syntax** | `COPY x REPLACING a BY b.` | `REPLACE ==a== BY ==b==.` |
| **Termination** | Ends when copybook inclusion ends | Ends with `REPLACE OFF.` or another `REPLACE` |
| **Processing order** | Applied first (during COPY expansion) | Applied after COPY expansion |

**REPLACE statement example:**

```cobol
       REPLACE ==:PROD:== BY ==WS-PRODUCT-==.

       COPY PRODREC.
       COPY PRODHIST.

       REPLACE OFF.
```

This applies the replacement to both copybooks and any intervening source text. It is sometimes used when multiple copybooks share the same placeholder convention.

**Processing order matters:** REPLACING on the COPY statement is applied first. Then, the REPLACE statement (if active) is applied to the result. This means they can be layered, but care must be taken to avoid unexpected interactions.

### Nested Copybooks

A copybook can itself contain COPY statements, resulting in nested inclusion. This is permitted by the COBOL standard, subject to compiler-defined limits on nesting depth.

**Example:**

Main program:
```cobol
       WORKING-STORAGE SECTION.
       COPY ALLRECS.
```

Copybook `ALLRECS`:
```cobol
       COPY CUSTREC.
       COPY ACCTREC.
       COPY TRANREC.
```

Copybook `CUSTREC`:
```cobol
       01  WS-CUSTOMER-RECORD.
           05  WS-CUST-ID         PIC 9(08).
           05  WS-CUST-NAME       PIC X(30).
```

The compiler expands `ALLRECS`, which triggers expansion of `CUSTREC`, `ACCTREC`, and `TRANREC` in turn.

**Rules for nested copybooks:**

- A copybook must not directly or indirectly copy itself (no recursive inclusion). The compiler will flag this as an error.
- REPLACING on an outer COPY does not propagate into nested COPY statements. Each COPY statement's REPLACING applies only to the text of the directly referenced copybook, not to text introduced by nested COPY statements within it.
- The REPLACE statement, however, does apply to text introduced by nested copybooks because it operates on the final expanded source stream.
- Compiler limits on nesting depth vary. IBM Enterprise COBOL supports nesting but does not specify a fixed limit in the language reference; practical limits depend on compiler resources.

### SUPPRESS Directive

Some compilers support a `SUPPRESS` option (or `SUPPRESS PRINTING`) on the COPY statement to prevent the copybook text from appearing in the compiler listing:

```cobol
       COPY CUSTREC SUPPRESS.
```

This is useful for large or frequently included copybooks where the listing would become unwieldy. The copybook is still fully included and compiled; only the listing output is affected.

Note: `SUPPRESS` is an IBM extension adopted by several other compilers. Its availability and exact syntax may vary.

## Common Patterns

### Record Layouts for Files

The most widespread use of copybooks is defining record layouts for files. The FD (File Description) entry is in the main program, but the record layout lives in a copybook:

```cobol
      * In the main program:
       FILE SECTION.
       FD  CUSTOMER-FILE
           RECORDING MODE IS F
           RECORD CONTAINS 150 CHARACTERS.
       COPY CUSTFDREC.

      * In the WORKING-STORAGE SECTION, a working copy:
       WORKING-STORAGE SECTION.
       01  WS-CUSTOMER-REC.
           COPY CUSTREC.
```

This pattern ensures that every program reading or writing the customer file uses the identical layout. When the file layout changes, only the copybook is updated, and all programs are recompiled.

### Database Record Descriptions (DCLGEN)

In DB2 environments, the DCLGEN utility generates copybooks containing host variable declarations that match database table columns:

```cobol
       WORKING-STORAGE SECTION.

      * DB2 DCLGEN output for CUSTOMER table
           EXEC SQL INCLUDE DCLCUST END-EXEC.
```

The generated copybook `DCLCUST` typically contains:

```cobol
      * DCLGEN TABLE(CUSTOMER)
      *        LIBRARY(DB2.DCLGEN.LIB(DCLCUST))
      *        LANGUAGE(COBOL)
      *        STRUCTURE(DCUST)
       01  DCUST.
           10  CUST-ID            PIC S9(9) COMP.
           10  CUST-NAME          PIC X(30).
           10  CUST-ADDR          PIC X(50).
           10  CUST-BAL           PIC S9(7)V9(2) COMP-3.
      * DCLGEN also generates null indicator structure
       01  DCUST-NULL.
           10  CUST-ID-NI         PIC S9(4) COMP.
           10  CUST-NAME-NI       PIC S9(4) COMP.
           10  CUST-ADDR-NI       PIC S9(4) COMP.
           10  CUST-BAL-NI        PIC S9(4) COMP.
```

Note: `EXEC SQL INCLUDE` is functionally similar to `COPY` but is processed by the DB2 precompiler rather than the COBOL compiler. The member is searched in the library specified by the `SYSLIB` DD for the DB2 precompile step.

### Constants and Configuration Values

Copybooks centralize constant definitions so that all programs use the same values:

```cobol
      * Copybook: APPCNST
       01  WS-APP-CONSTANTS.
           05  WS-MAX-RETRY       PIC 9(02) VALUE 03.
           05  WS-PAGE-SIZE        PIC 9(03) VALUE 060.
           05  WS-APP-VERSION      PIC X(08) VALUE '02.05.01'.
           05  WS-COMPANY-NAME     PIC X(30) VALUE 'ACME CORPORATION'.

       01  WS-STATUS-CODES.
           05  WS-STATUS-ACTIVE    PIC X(01) VALUE 'A'.
           05  WS-STATUS-INACTIVE  PIC X(01) VALUE 'I'.
           05  WS-STATUS-CLOSED    PIC X(01) VALUE 'C'.
           05  WS-STATUS-PENDING   PIC X(01) VALUE 'P'.
```

### Communication Area Layouts

In CICS and batch-to-batch program communication, the communication area (COMMAREA or LINKAGE SECTION layout) is defined in a copybook so that both the calling and called programs agree on the structure:

```cobol
      * Copybook: CUSTCOMM
      * Communication area for CUSTINQ program
       01  CUST-COMMAREA.
           05  COMM-FUNCTION       PIC X(01).
               88  COMM-INQUIRY        VALUE 'I'.
               88  COMM-UPDATE         VALUE 'U'.
               88  COMM-DELETE         VALUE 'D'.
           05  COMM-CUST-ID        PIC 9(08).
           05  COMM-RETURN-CODE    PIC 9(02).
               88  COMM-SUCCESS        VALUE 00.
               88  COMM-NOT-FOUND      VALUE 10.
               88  COMM-ERROR          VALUE 99.
           05  COMM-CUST-DATA.
               10  COMM-CUST-NAME PIC X(30).
               10  COMM-CUST-ADDR PIC X(50).
```

Both the calling program and the called program include:

```cobol
       LINKAGE SECTION.
       COPY CUSTCOMM.
```

### Paragraph Includes

Although less common than data copybooks, some shops use copybooks to include standard procedure-division paragraphs:

```cobol
       PROCEDURE DIVISION.
       MAIN-LOGIC.
           PERFORM INITIALIZATION
           PERFORM PROCESS-RECORDS
           PERFORM CLEANUP
           STOP RUN.

       COPY INITPARA.
       COPY CLEANPAR.
```

Where `INITPARA` might contain:

```cobol
       INITIALIZATION.
           OPEN INPUT CUSTOMER-FILE
           OPEN OUTPUT REPORT-FILE
           INITIALIZE WS-COUNTERS
           MOVE FUNCTION CURRENT-DATE TO WS-RUN-DATE.
```

This pattern is used when multiple programs share identical startup or teardown logic. However, many shops discourage this practice because it makes the PROCEDURE DIVISION harder to read and debug. Shared logic in the PROCEDURE DIVISION is more commonly implemented via CALL to a subprogram rather than COPY of paragraphs.

### Condition-Name Tables

Copybooks are ideal for defining sets of 88-level condition names that decode status codes or flags:

```cobol
      * Copybook: SQLCODES
       01  WS-SQLCODE-AREA.
           05  WS-SQLCODE          PIC S9(09) COMP.
               88  SQL-SUCCESS         VALUE 0.
               88  SQL-NOT-FOUND       VALUE 100.
               88  SQL-DUP-KEY         VALUE -803.
               88  SQL-DEADLOCK        VALUE -911.
               88  SQL-TIMEOUT         VALUE -913.
               88  SQL-RESOURCE-BUSY   VALUE -904.
```

### Prefixed Copybooks with REPLACING

A powerful and widespread pattern is writing copybooks with placeholder tags and using REPLACING to customize each inclusion. This allows a single copybook to define a record layout that is included multiple times with different prefixes:

```cobol
      * Copybook: ADDRLAY (generic address layout)
       05  :TAG:-STREET            PIC X(40).
       05  :TAG:-CITY              PIC X(25).
       05  :TAG:-STATE             PIC X(02).
       05  :TAG:-ZIP               PIC X(10).
       05  :TAG:-COUNTRY           PIC X(03).
```

Used in a program:

```cobol
       01  WS-HOME-ADDRESS.
           COPY ADDRLAY REPLACING ==:TAG:== BY ==HOME==.

       01  WS-WORK-ADDRESS.
           COPY ADDRLAY REPLACING ==:TAG:== BY ==WORK==.

       01  WS-MAIL-ADDRESS.
           COPY ADDRLAY REPLACING ==:TAG:== BY ==MAIL==.
```

This expands to three distinct address structures with prefixed names (`HOME-STREET`, `WORK-STREET`, `MAIL-STREET`, etc.), all guaranteed to have identical field definitions.

Common placeholder conventions include:
- `:TAG:` or `:PREFIX:` (colon-delimited to avoid accidental matches)
- `@PREFIX@` (at-sign delimited)
- `DFH-` or `XXX-` (a dummy prefix meant to be replaced)

## Gotchas

- **REPLACING does not cascade into nested COPY statements.** If copybook A contains `COPY B`, and you write `COPY A REPLACING ==X== BY ==Y==`, the replacement applies only to text physically in A, not to text brought in by the nested COPY B. Use the REPLACE statement instead if you need replacements to cascade through nested expansions.

- **Pseudo-text matching is whitespace-sensitive in some compilers.** The COBOL standard specifies that sequences of whitespace are treated as a single space for matching purposes, but some older compilers or non-standard implementations may be more literal. Always test REPLACING patterns after compiler upgrades.

- **Missing period at the end of the COPY statement.** The COPY statement must be terminated with a period. Forgetting it causes cryptic compiler errors because the compiler tries to parse the next line as part of the COPY syntax. Example: `COPY CUSTREC` without the trailing period.

- **Copybook context dependency.** A copybook that works perfectly in WORKING-STORAGE might cause errors if included in the FILE SECTION or LINKAGE SECTION, because valid clauses differ between sections. For example, VALUE clauses on level-01 through level-49 items are not allowed in the FILE SECTION in standard COBOL. Document where each copybook is intended to be used.

- **REPLACING with partial word matches can be dangerous.** If you use pseudo-text replacing like `==REC== BY ==RECORD==`, it will match `REC` anywhere it appears as a separate word, but not inside compound words (pseudo-text respects word boundaries when the operand is a complete word). However, if the pseudo-text contains special characters or hyphens, matching behavior can be unexpected. Always use distinctive placeholder tags like `:TAG:` to avoid accidental matches.

- **Circular or recursive COPY references.** If copybook A copies B and copybook B copies A, the compiler will detect the cycle and report an error. This is usually caused by refactoring mistakes. Check the full chain of nested copybooks before adding a new COPY statement.

- **Copybook not found errors from library search order.** When multiple libraries are concatenated, the compiler uses the first matching member name. If a copybook with the same name exists in multiple libraries, you may silently pick up the wrong version. Use the IN/OF clause to be explicit when ambiguity is possible, or carefully manage your library concatenation order.

- **Tabs in copybooks.** Some compilers do not handle tab characters correctly in copybooks, particularly in fixed-format (columns 7-72) source. A tab that looks correct in an editor may shift columns unexpectedly when the compiler processes it. Use spaces, not tabs, in copybooks.

- **Forgetting to recompile all dependent programs.** When you change a copybook, every program that includes it must be recompiled. Missing even one program creates a version mismatch that can cause data corruption at runtime. Use build tools, dependency trackers, or make-style systems to track copybook dependencies.

- **EXEC SQL INCLUDE vs COPY.** These are processed by different components (DB2 precompiler vs COBOL compiler) and search different libraries. Do not confuse them. A DCLGEN member referenced via `EXEC SQL INCLUDE` must be in the DB2 precompiler's SYSLIB, while a member referenced via `COPY` must be in the COBOL compiler's SYSLIB.

- **Level numbers in copybooks.** A copybook defining subordinate items (05, 10, etc.) without a 01-level group expects the including program to provide the 01-level. If the including program's 01-level has a different structure than expected, fields may be misaligned. Clearly document whether a copybook provides a complete 01-level or only subordinate items.
